The one-line version: a candidate first survives 3 yes/no knock-outs (identical to being economically comparable at all), then earns a 0–1 score on 5 weighted dimensions, then the top 15 scores are kept but weighted by how close the match actually is — so a borderline peer counts for less than an exact one, rather than being treated as equal or thrown away.

1End-to-end flow

From the full candidate universe to the priced peer set, in order.

1

Universe load

Every company in the database except the target itself is fetched and normalized into a Company record, then classified into an EconomicProfile (operating model, value chain, customer type).

2

Stage B — hard knock-outs

Each candidate is checked against 3 yes/no rules. Any mismatch = immediate rejection, with a logged reason. No scoring happens for a rejected candidate.

3

Stage C — similarity scoring

Survivors are scored 0–1 on 5 weighted dimensions. Scores are summed and the list is sorted descending.

4

Stage D — post-scoring controls

The top 15 scores are kept. Each peer's contribution is then weighted by match quality, its multiple made eligible only if listed/market-priced, and outlier multiples trimmed before the final valuation math runs.

5

Priced peer set

What remains is the peer set — with a per-peer weight — that the valuation engine actually uses.

STAGE A · ELIGIBILITY

2Who even enters the pool

Two checks shape the candidate pool before any comparison begins.

#CheckRuleWhy
1Geography countryISOAlpha2Code = "IN" The search is scoped to India at the D&B company_search call — the universe never contains a foreign company to begin with.
2Self-exclusion peer.duns != target.duns The target cannot be its own comparable. Removed before the universe loop runs.
STAGE B · HARD KNOCK-OUTS

3Reject before scoring

Three checks, evaluated in this order, in core/pipeline.py :: discover_peers. The first one that fails produces the rejection reason — the rest are not even checked once one fails.

Why hard knock-outs at all, instead of just scoring everything? A distributor, retailer, or service firm is not a manufacturing comparable no matter how similar its revenue or margin look — comparing a manufacturer to a wholesaler on "5% weaker scale-match" would be nonsensical, not just imprecise. These three checks encode that judgment as a hard rule rather than a soft penalty.

#CheckPasses whenReal rejection (Woodward run)
3Operating model peer.operating_model == target.operating_model ✕ reject Metro Wholesale Distributors — distributor vs manufacturer
4Value chain peer.value_chain == target.value_chain ✕ reject Sterling Steel Billets — raw_material vs finished_goods (it IS a manufacturer, but of the wrong value-chain position)
5Major industry peer.major_industry == target.major_industry D&B major code: D = Manufacturing, F = Wholesale Trade, G = Retail Trade, I = Services

STAGE C · WEIGHTED SIMILARITY SCORE

4How survivors are ranked (0–1 score)

Every candidate that survives Stage B gets a score — five dimensions, each independently rated 0–1 and multiplied by a fixed weight. The weights sum to exactly 1.00.

# dimension = raw(0..1) × weight industry 0.40 = 1.0 if same NAICS 3-digit subsector 0.6 if same D&B Hoovers class (subsector differs) 0.0 otherwise scale 0.20 = 1 / (1 + |log1p(rev_target) − log1p(rev_peer)|) margin 0.15 = max(0, 1 − 5 × |margin_target − margin_peer|) customer 0.15 = 1.0 if same customer_type (B2B/B2C/mixed) else 0.0 export 0.10 = 1.0 if same exporter flag else 0.3 (never 0 — a shared industry still counts) score = Σ(dimension) # range 0.0 – 1.0
#DimensionWeightWhat it checks
6Industry proximity0.40 Same NAICS 3-digit subsector scores highest; falling back to the same broader Hoovers class scores partial credit; otherwise zero. Heaviest weight — the single biggest driver of the final score.
7Scale proximity0.20 Revenue closeness on a log scale, so a ₹200 Cr vs ₹400 Cr gap counts more than a ₹5,000 Cr vs ₹5,200 Cr gap of the same absolute size — matches how investors actually think about scale.
8Profitability proximity0.15 EBITDA-margin closeness; the 5× multiplier means a 20-point margin gap already zeroes this dimension out.
9Customer type0.15 B2B vs B2C vs mixed — binary match, no partial credit.
10Export profile0.10 Same exporter/domestic status; a mismatch still scores 0.3, not 0 — the lightest-weight, softest check. This is the only trade-related signal in the entire system.
STAGE D · POST-SCORING QUALITY CONTROLS

5What happens to the ranked list before it reaches the math

Being in the top 15 by score is not the end of the filtering — what happens next determines how much each surviving peer actually influences the final number.

#ControlRuleEffect
11Top-N cut top_n = 15 Only the 15 highest-scoring survivors are carried into valuation (the full ranked list is still recorded, for transparency — e.g. "15 of 53 ranked").
12Similarity weighting weight = clamp((score−0.40)/(0.85−0.40), 0.15, 1.0) Score ≥ 0.85 → full weight 1.0. Below that it tapers linearly to a 0.15 floor. A peer that barely made the top 15 (say score 0.62) still counts, but at roughly half strength — it cannot swing the answer as much as an exact match. This is the direct answer to "what if there aren't enough exact peers": the borderline ones are automatically down-weighted rather than either excluded or treated as equal.
13Multiple eligibility listed + market EV preferred; book capital-employed fallback if <3 listed comps; driver > 0 Only listed peers with an observable market enterprise value feed a true trading multiple. Unlisted peers still inform the similarity scoring but contribute no multiple unless the book-value fallback triggers.
14Outlier trim Tukey fence: drop values outside [Q1 − 1.5·IQR, Q3 + 1.5·IQR] Applied per valuation method on the (weighted) peer multiples, before they are read at the target's quality position. Removes freak multiples without a manual override.
Net effect of Stage D: the headline number is protected on two independent fronts — weighting softens the influence of imperfect matches, and trimming removes statistical outliers. Neither alone would be enough; together they mean a thin or imperfect peer set degrades the estimate's precision (wider range, lower confidence) rather than silently distorting it.

8What is explicitly NOT used

To prevent wasted effort or false assumptions about the model.

Import data — not used. Not present in the D&B financial payload the engine reads.
Per-capita metrics — not used. No population/regional data enters the pipeline at all.
The only trade-related signal is the export flag (filter #10, weight 0.10, soft mismatch = 0.3 not 0). That is the complete list — nothing else related to trade or geography beyond the India-only search scope (filter #1) feeds the filtering or scoring.
